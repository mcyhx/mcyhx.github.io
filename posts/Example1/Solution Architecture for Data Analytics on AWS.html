1. Introduction

Legacy analytics platforms struggle with fast-growing datasets, restricting companies from unlocking the full potential of their data. This post explores how to build a modern, scalable analytics architecture on AWS leveraging services like S3, Athena, Amazon Kinesis Data Firehose, Amazon Athena, and Amazon QuickSight. We'll discuss an example of a software company that needs a data analytics solution in AWS so that it can derive insights from clickstream data.

3. Architecture Solutions Designing

When designing architecture solutions, it is crucial to use the right tools to address the customer's specific requirements and needs. The job at hand is satisfying what the customer asks for in their scenario.

Throughout the design process, communicating with the customer is very important, as requirements often need further clarification or the customer's situation requires additional investigation. Open dialogue allows for collaboratively ensuring the architecture meets the customer's evolving needs and technical objectives.

Data Ingestion

  1. Customer’s situation:

The customer primarily needed a way to ingest large volumes of clickstream data that would scale, without having to change their existing JavaScript library. They wanted to avoid the overhead of managing underlying resources, with costs aligned to usage.

To address this, we integrated API Gateway. This allowed us to seamlessly bring in the customer's current codebase while providing a sturdy, secure HTTPS data ingestion mechanism into the new architecture.

Customers require scalable data ingestion for clickstream data without having to modify their current JavaScript library or manage resources while keeping costs proportional to usage.

  1. Different data ingestion tools features:
  • EMR: Managed Hadoop clusters for big data streaming with Spark
  • DMS: Migrates databases to the cloud
  • Data Exchange: Provides data catalogs and third-party data integration
  • Kinesis Data Analytics: Real-time data processing during ingestion
  • Kinesis Data Streams: Lower latency but needs custom coding
  • Kinesis Firehose: More convenient with buffering, no custom coding
  1. Kinesis Firehose was chosen because:

Kinesis Firehose’s convenience as a managed service, meeting the customer's needs of asynchronous analytics without low latency requirements or custom coding.

Data Storage

  1. Customer’s situation:

The customer requires a storage solution for growing volumes of analytics data that is durable, encrypted, replicated, cost-efficient, serverless, and flexible for use with different processing tools. S3 meets all these needs for a scalable, resilient data lake to power their analytics application.

  1. Different storage tools features:
  • EBS: Block storage, high performance, attaches to EC2 instances
  • EFS: File storage, higher durability, attaches to EC2, containers
  • S3: Object storage, highest durability, standalone service
  1. Choose S3 was chosen because:

S3 was chosen over EBS and EFS because of its standalone storage capabilities, granular usage pricing, and ability to meet all the durability, encryption, replication, and cost requirements.

Data Processing and Analysis

  1. Customer’s situation:

The customer has clickstream data files in S3 in JSON format that need to be analyzed with SQL. They want a serverless service with pay-per-query pricing and without the overhead of managing servers or clusters.

  1. Different data processing tools features:
  • S3 Select: SQL queries on individual S3 objects
  • Glue: Managed ETL jobs, good for unstructured data
  • EMR: Managed Hadoop framework for big data processing
  • Athena: Interactive SQL queries across S3 data, serverless
  1. Athena was chosen because:

Athena serverless, pay-per-query model that can directly query formats like JSON in S3 with standard SQL, avoiding complexity and matching the customer needs. Athena meets these requirements for convenience, simplicity, and an aligned cost model.

Data Visualization

  1. Customer’s situation:

The customer requires data visualization to surface insights, create interactive dashboards for their clients, and communicate data findings visually. This complements the underlying SQL queries and enables deeper business analytics. The customer uses QuickSight in their BI team.

  1. Different data visualization tools features:
  • CloudWatch: Metrics and dashboards for monitoring AWS resources
  • OpenSearch: Managed Elasticsearch service with embedded storage/compute
  • Grafana: Open source platform for visualizing time series metrics
  • QuickSight: Serverless BI tool with pay-per-use pricing
  1. QuickSight was chosen because:

QuickSight was suitable for the task and the customer already uses it for other projects, so adopting it for this architecture requires less effort and training. Its serverless model and Athena integration also align with the goals of this analytics solution.

Proposed Architecture

Leveraging API Gateway's integration capabilities and security features, we can provide a robust HTTPS endpoint that seamlessly ingests the customer's existing code payload while securely proxying the requests to Kinesis Firehose – all without requiring front-end code modifications.

  • API Gateway - ingest data from HTTPS requests
  • Kinesis Firehose - scalable ingestion to S3
  • S3 - durable object storage
  • Athena - serverless SQL queries on S3 data
  • QuickSight - visualize query results and insights

4. Optimization and Scalability

  • Use a serverless architecture instead of EC2 which the customer currently leverages for their menu update system
  • Use CloudFront CDN in front of S3 web hosting to cache content and reduce requests/costs
  • Replace API Gateway with Amazon Cognito for authentication. Remove API Gateway to reduce costs. Update JavaScript code to integrate Cognito
  • Create a CloudFormation template of the architecture to reproduce it in other accounts. Helps with governance, and billing separation
  • Add retry logic and exponential backoff to the JavaScript library for better resilience. Leverage built-in retries in AWS SDK
  • Optimize S3 data layout for Athena with partitioning and columnar formats like Parquet. Improves query performance

In summary, the key optimization tips covered transitioning to serverless, using CloudFront for caching, replacing API Gateway with Cognito, CloudFormation infrastructure as code, resilient JavaScript clients, and tuning S3 for fast Athena queries.
